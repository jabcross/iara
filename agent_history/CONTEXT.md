# Iara Environment Setup - Technical Context

## Overview

This session restructured the iara development environment to separate concerns across
three package management layers: **spack** (native libraries), **modules** (system tools),
**pip/venv** (Python packages), and **nvm** (Node.js).

## Architecture

### Environment loading order

Everything is initiated by `source sorgan_env.sh`:

1. **Local spack** (`iara/spack/`, v1.1.1) is sourced first, overriding the global spack
   at `/export/spack` (v0.24.0) that is auto-loaded via `/etc/profile.d/spack.sh`.
   The `iara_env` spack environment is activated.
2. **Modules** are loaded (gdb, gcc, ninja, cmake, ccache, casacore, abseil-cpp, mold,
   valgrind, python/3.13.1, openblas).
3. **nvm** is sourced and pinned to Node.js v25.6.1.
4. **Python venv** at `.venv/` is activated (created on first run with packages from
   `requirements.txt`).
5. **`.env`** is sourced, setting project paths (`IARA_DIR`, `LLVM_SOURCES`, etc.) and
   spack view paths (`SPACK_VIEW_PATH`, `LD_LIBRARY_PATH`, etc.).

### Package management split

| Layer | What it provides | Config file |
|-------|-----------------|-------------|
| Spack (`sorgan_spack.yaml`) | `gcc-runtime@15`, `fftw@3` | `sorgan_spack.yaml` |
| Modules (from global spack) | gcc, gdb, cmake, ninja, ccache, mold, valgrind, python, openblas, casacore, abseil-cpp | loaded in `sorgan_env.sh` |
| pip/venv | Python packages (numpy, matplotlib, pandas, jupyter, etc.) | `requirements.txt` |
| nvm | Node.js v25.6.1, npm | loaded in `sorgan_env.sh` |

**Rationale**: Most build tools are already available as modules from the global spack
installation. Python packages are handled by pip for speed on x86 (binary wheels).
Only `gcc-runtime@15` and `fftw@3` remain in the local spack because they're not
available as modules.

**Cross-architecture note**: A non-x86 target architecture may require building Python
packages from source. In that case, the `py-*` packages should be added back to
`sorgan_spack.yaml` instead of using `requirements.txt`/venv.

### Spack upstream configuration

`sorgan_spack.yaml` configures the global spack installation at `/export/spack/opt/spack`
as an **upstream**. With `concretizer.reuse: true`, the local spack reuses packages already
built globally rather than rebuilding them. All current specs (`fftw@3.3.10`,
`gcc-runtime@15.2.0` and their dependencies) are fully satisfied from the upstream.

### File relationships

- **`sorgan_spack.yaml`** is the single source of truth for spack configuration.
- **`spack.yaml`** (repo root) is a **symlink** to `sorgan_spack.yaml`.
- **`iara_env/spack.yaml`** (inside `spack/var/spack/environments/`) is also a
  **symlink** to `sorgan_spack.yaml`.
- **`.env`** is generated by `scripts/generate_env.sh`. Do not edit it directly.
- **`sorgan_env.sh`** is the main entry point. Source it to set up the full environment.

### Local spack version

The local spack installation at `iara/spack/` was updated from v0.24.0.dev0 to **v1.1.1**
(`releases/latest` tag). This was required because the old version couldn't read v6 format
lockfiles.

## Known issues

### Global spack conflict (cosmetic, affects Claude Code only)

The system-wide `/etc/profile.d/spack.sh` loads the global spack v0.24.0 on every new
shell. When `SPACK_ENV` is set (pointing to `iara_env`), the global spack tries to read
the v6 lockfile and emits:

```
==> Error: Spack 0.24.0.dev0 cannot read the lockfile ... using the v6 format.
```

This error is **cosmetic** in the user's shell (the local spack overrides it via
`sorgan_env.sh`). However, Claude Code's Bash tool spawns new shells that re-source
`/etc/profile.d/spack.sh`, so the error appears in tool output. It does not affect
command execution.

**Attempted fix**: A Claude Code `SessionStart` hook using `CLAUDE_ENV_FILE` was tried
but `CLAUDE_ENV_FILE` only supports `export KEY=VALUE` syntax and cannot run `source`
commands or reliably override variables set by `/etc/profile.d/`.

**Proper fix**: Update the global spack at `/export/spack` to v1.1.1 (requires sysadmin).

### PATH corruption from .bashrc (fixed)

`.bashrc` previously appended a nix profile script path as a literal PATH entry:
```
PATH=...:$PATH:/scratch/$USER/nix/var/nix/profiles/default/etc/profile.d/nix.sh
```
This corrupted PATH by concatenating entries without `:` separators, preventing nvm from
working. The nix entry was removed.

## Unfinished work

1. **Cross-architecture Python packages**: The `requirements.txt`/venv approach only works
   well on x86 where pip has binary wheels. For a non-x86 target, the `py-*` packages need
   to go back into `sorgan_spack.yaml`. A conditional mechanism (per-architecture config or
   conditional logic in `sorgan_env.sh`) has not been implemented.

2. **`setup-environment.sh` cleanup**: The script still has redundant spack view path
   exports (lines 76-82) that duplicate what `.env` already provides. The commented-out
   Polygeist sections could also be removed.

3. **Global spack upgrade**: The `/export/spack` installation should be updated to v1.1.1
   to eliminate the lockfile format error. This requires sysadmin access.

4. **Nothing has been committed**. All changes from this session are unstaged.

## Quick start for a new developer

```bash
# From the iara project root:
source sorgan_env.sh

# This sets up everything:
#   - Local spack (v1.1.1) with iara_env active
#   - System modules (gcc 14.2.0, cmake, ninja, etc.)
#   - Node.js v25.6.1 via nvm
#   - Python 3.13.1 venv with scientific packages
#   - Project paths (LLVM_SOURCES, LLVM_INSTALL, etc.)
```

For a clean-slate setup (clones spack, LLVM, builds everything):
```bash
sh scripts/setup-environment.sh
```
